## Исследование lock_lost_grace

### Тест: lock_lost_grace=2 (+ night_confirm=4)

**Гипотеза:** восстановление lock_lost_grace с 1 до default 2 даёт large drone
1 дополнительный кадр толерантности перед recovery scan → меньше bursty release.

**Результат:**
- `night_ground_large_drones`: false_lock=0.604 (↓ от 0.771), **id_chg=55.05** (↑ регрессия!)
- `night_ground_indicator_lights`: false_lock=0.237, id_chg=0.00 (PASS!)

**Вывод:** lock_lost_grace=2 вызвал id_chg регрессию. При grace=2 система ждёт дольше перед
recovery scan → к моменту scan дрон уместился дальше → spatial gate промахивается →
другой target подхватывается → ID change. lock_lost_grace=1 — правильный выбор.

- `night_ground_indicator_lights`: false_lock=0.243, id_chg=0.00 (PASS!)

**Вывод:** night_confirm=4 — key driver обоих улучшений. lock_lost_grace остаётся на 1.

### Тест: night_confirm=5, lock_lost_grace=1

**Результат:**
- `night_ground_large_drones`: **pass=True**, false_lock=0.510, id_chg=12.23
- `night_ground_indicator_lights`: **pass=True**, false_lock=0.096, id_chg=0.00
- Night problem-pack gate: **PASS** (exit code 0, первый раз в истории проекта)

**Механизм:** night_confirm=5 означает, что новая night detectionтребует 5 последовательных
кадров. Transient false positives (как мигающие огни), которые существуют 1-3 кадра,
перестают регистрироваться как детекции. Это радикально снижает:
1. false_lock: ложные night targets не становятся tracked candidates
2. id_chg: ложные candidates не могут вытеснить существующий large-drone track
