## Executive Summary

GimbalProject находится в зрелом состоянии для проекта трекинга БПЛА с хорошо определёнными архитектурными принципами, работающим quality-gate и систематическим orchestration workflow. За последние три AP-цикла (001..009) выполнено 31 task — рефакторы, UI-задачи, hardening, evaluation. Однако аудит выявил три P0-находки, требующие немедленной attention: отсутствующий исходный файл UI-темы, расхождение между documented и actual filesystem state рефакторов, и persistent незакрытый вопрос о следующем training cycle после отклонения epoch142.

## 1. Architecture Health

### Сильные стороны
- **Backend-first design соблюдается** — `src/uav_tracker/` изолирован от UI. `app/` содержит только thin wrappers и GUI thread.
- **Multi-runtime abstraction работает** — `runtime/base.py` + `ultralytics_backend.py` + `hailo_backend.py`. Одна pipeline логика, два target environment.
- **5 пресетов охватывают операционные сценарии** — day / night / small_target / antiuav_thermal / rpi_hailo. Каждый пресет — complete context (model, thresholds, display, budget).
- **Lock policy изолирована** — `tracking/target_manager.py` + `tracking/lock_tracker.py`. Pipeline не принимает lock-decisions напрямую.
- **State machine canonical** — SCAN/TRACK/LOST — единственный UI source of truth через `TrackerWorker`.

### Архитектурные риски
- `app/main_gui.py` ~90KB — UI-монолит. Содержит TrackerWorker, MainWindow, build_*, _wire_actions, _update_stats в одном файле. Параллельно с pipeline.py — два крупных монолита.
- Нет чёткого ownership для `auto scene detection` — логика `_adapt_auto_scene()` в `pipeline.py`, параметры в `config.py`, но preset-специфичного переопределения нет (только global AUTO_SCENE_* поля).

## 2. Filesystem Audit — Расхождения Docs vs Code

### P0-1: `theme.py` source file ОТСУТСТВУЕТ

| Факт | Значение |
|------|----------|
| Ожидалось | `app/ui/theme.py` |
| Реально | `app/ui/__pycache__/theme.cpython-311.pyc` (только байткод) |
| Риск | При смене Python ≥ 3.12, или на другом Mac с Python 3.11+ без совместимого magic number — тема становится невосстановимой |
| Action | **Восстановить исходный файл** из `.pyc` (decompile) или создать новый `theme.py` с APP_STYLESHEET, который используется в `main_gui.py` |

## 4. Evaluation & Quality Gate

### Текущий Baseline (на дату аудита)

| Метрика | Baseline Value | Порог quality-gate |
|---------|---------------|-------------------|
| avg_fps | 85.2 | ≥ 8.0 |
| false_lock_rate | 0.724 (aggregate) / 0.861 (night) | ≤ 0.35 (aggregate) / ≤ 0.18 (noise) |
| active_id_changes_per_min | varies | ≤ 8.0 |

**Важно:** `false_lock_rate = 0.724` превышает quality-gate порог `--max-false-lock-rate 0.35` в более чем 2 раза. Это означает, что текущий baseline сам по себе не прошёл бы строгий quality-gate по этой метрике. TASK-029 (false-lock hardening) принят — но снижение требует верификации через повторный smoke после внесённых изменений.

### Regression Pack Coverage

`configs/regression_pack.csv` содержит 6 фиксированных клипов: 1 day, 2 night, 2 IR, 1 noise.

**Gap:** Нет preset-специфичных regression pack. Пресет `night.yaml` и `antiuav_thermal.yaml` имеют существенно разные параметры, но тестируются одним и тем же набором. При тюнинге `night.yaml` изменения могут быть скрыты агрегатом.

**Рекомендация:** Создать per-preset regression pack (~2 clip/preset) или добавить `scene` тег в CSV и запускать quality-gate с `--scene night` / `--scene ir` фильтрами.

## 5. Infrastructure

### Model Paths — Не Version-Locked (P1)

Все 5 пресетов содержат пути типа:
```yaml
model_path: "runs/detect/runs/drone_bird_probe_fast/weights/best.pt"
```

Этот путь — продукт локального `runs/` директории. На другом Mac или в чистом checkout — модели нет.

**Риски:**
- CI не может запустить полный quality-gate без local model
- Новый collaborator или машина — нет воспроизводимости
- После clean rebuild `runs/` (обычная практика при disk cleanup) — пресеты сломаны

**Action:** Publish baseline model в GitHub Releases или artifacts storage. Добавить в preset `model_version: "v1.0-baseline"` и скрипт для загрузки.

### requirements.txt — Отсутствует (P1)

Зависимости хранятся только в `tracker_env/` venv. Установленные пакеты (80+):
- torch 2.10.0, torchvision 0.25.0
- ultralytics 8.4.19
- PySide6 6.10.2
- opencv-python 4.13.0.92
- numpy 2.4.2

Нет `requirements.txt` или `pyproject.toml` в корне проекта.

**Action:** `pip freeze > requirements.txt` + trim до production-only deps. Или использовать `pip-compile` с `requirements.in`.

### CI Pipeline — Облегчённый (P2)

`.github/workflows/ci.yml` выполняет только:
1. Python syntax check (`compileall`) на src/, app/, python_scripts/
2. Orchestration state consistency check

**Не включает:**
- Запуск `tests/` (pytest/unittest)
- Quality-gate (требует видео-файлы, не подходит для CI)
- Lint/type check (ruff, mypy)

**Action (P2):** Добавить в CI шаг `python -m pytest tests/ -q` — это займёт <5 секунд, покрывает 12 существующих тестов и вернёт мгновенную обратную связь при lock-policy регрессии.

### Training Conveyor — Tight Coupling (P2)

Automation via `automation/` + GitHub manifests (RTX → Mac intake). Текущая схема:
- RTX создаёт ZIP-артефакт, загружает на GitHub
- Mac intake скрипт забирает по манифесту
- Разворачивает в `imports/rtx/unpacked/`

**Single point of failure:** Corrupt manifest → потеря трекинга артефакта. Нет checksum validation.

**Action (P2):** Добавить SHA-256 checksum в manifest JSON и validation в `fetch_training_artifact.py`.
