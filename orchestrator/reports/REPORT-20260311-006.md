## Описание утилиты
`validate_profile_presets.py` сканирует `configs/*.yaml` (или явно указанные файлы) и проверяет:

| Проверка | Описание |
|---|---|
| Unknown keys | Ключи YAML, отсутствующие в маппинге `profile_io.apply_overrides` |
| Type mismatch | Значения с неверным типом (bool/int/float) относительно правил `_TYPE_RULES` |
| Non-flat skip | Файлы с вложенными dict/list пропускаются (не являются пресетами) |

Ключевая особенность: `extract_mapping_keys()` парсит `profile_io.py` через `ast` без импорта модуля — нет зависимости от cv2/ultralytics/yaml при самом извлечении ключей. PyYAML нужен только для загрузки YAML-файлов.

Exit code: 0 — всё OK, 1 — есть ошибки, 2 — критическая ошибка (файл не найден).

## Residual Risks
- Risk: `_TYPE_RULES` в скрипте — дублирует знания о типах из `Config`. Если тип поля изменится в `Config`, правила нужно обновлять вручную.
- Risk: `extract_mapping_keys()` ищет первую `mapping = {...}` в файле по имени. Если переменная будет переименована или перестроена, ключи перестанут извлекаться.
- Risk: `training_curriculum.yaml` корректно пропускается как non-flat, но другие нестандартные плоские YAML-файлы в configs/ будут проверяться на известные ключи (и будут падать с unknown keys).

## Варианты эволюции
1. **Интеграция в CI** — добавить шаг `validate_profile_presets.py --configs-dir configs` в pre-commit hook или в `run_stable_cycle.py` перед запуском бенчмарка.
2. **Автоматический тип из Config** — читать дефолтные значения из `Config` датакласса через `dataclasses.fields()` и вывести типы динамически, исключив дублирование `_TYPE_RULES`.
3. **Диапазонная валидация** — проверять числовые поля на допустимые диапазоны (например, `conf_thresh` ∈ [0, 1], `imgsz` кратно 32).
4. **JSON-отчёт** — флаг `--output-json PATH` для machine-readable вывода (аналог `analyze_lock_events.py`), удобно для оркестратора.
