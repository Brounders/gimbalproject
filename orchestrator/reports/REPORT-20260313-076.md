1. **"Не training first"** — корректно. Epoch142 показал structural regression хуже baseline.
   Перезапуск обучения без понимания текущего runtime состояния преждевременен.

## Где Codex ошибается

**Ключевая логическая ошибка: stage-5 предполагает, что stage-4 не решил проблему.**
Но мы этого не знаем — пост-hardening измерений нет.

После AP-022 система находится в одном из трёх состояний, которые неразличимы без измерений:

| Состояние | false_lock после AP-022 | Вывод |
|-----------|------------------------|-------|
| Hardening работает хорошо | ~0.45–0.55 | Stage-5 не нужен; нужна финализация + training |
| Hardening частичный | ~0.60–0.70 | Stage-5 обоснован, но нужен конкретный evidence-based knob |
| Over-constrained система | ~0.80+ (близко к baseline) | Stage-5 ухудшит, нужна другая стратегия |

**Knob saturation risk:** 13 параметров изменены в одном направлении (всё "строже").
Чрезмерно строгая конфигурация:
- Дробит legitimate треки → снижает continuity_score (не отражается в false_lock_rate)
- Задерживает lock до степени, неприемлемой для оператора
- Создаёт false negatives (пропущенные реальные цели), невидимые в текущих метриках

Ни один из отчётов AP-018–022 не содержит post-hardening измерений continuity или presence.

**Aspirational ≠ achieved:** `max_false_lock_rate: 0.55` в contract — это цель, не факт.
Единственное измеренное значение: 0.861 (2026-03-11). Разрыв — 36 процентных пунктов.

## Наиболее рациональный следующий шаг

**НЕ stage-5. Измерение текущего состояния.**

```bash
PYTHONPATH=src python python_scripts/run_problem_pack_gate.py
```

Это даст первые post-hardening KPI после 5 слоёв изменений.

Затем по результату:
- **Gate PASS** (false_lock ≤ 0.55, id_chg ≤ 18/мин) → hardening работает. Следующий шаг:
  полный regression pack + рассмотрение нового training cycle с текущим baseline.
- **Gate FAIL, false_lock 0.55–0.70** → частичное улучшение, обоснованный stage-5 с одним-двумя
  конкретными knob, исходя из того, какой threshold фэйлит.
- **Gate FAIL, false_lock > 0.80** → hardening малоэффективен. Другой подход:
  night detector параметры (NIGHT_MAX_AREA, NIGHT_LOST_MAX — сейчас не в YAML mapping),
  или коррекция данных для нового training cycle.
