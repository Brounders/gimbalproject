# ENGINEERING CODEX — GimbalProject

Этот документ фиксирует инженерные принципы проекта и обязателен для всех агентных сессий.

## Базовые принципы

- Минимальные точечные изменения лучше крупных переписываний.
- Сначала стабилизация, потом упрощение, потом расширение.
- Любая новая сложность должна быть оправдана измеримой пользой.
- UI не должен содержать бизнес-логику.
- Бизнес-логика не должна зависеть от слоя отображения.
- Аппаратный доступ и IO должны быть изолированы.
- Любое изменение должно быть проверяемым.
- Не переписывать систему с нуля без необходимости.
- Сначала локальные исправления, затем постепенное улучшение архитектуры.

## Источник состояния

- Все агенты взаимодействуют через GitHub.
- Репозиторий является единственным источником состояния проекта.
- Любые решения и статусы фиксируются коммитами и markdown-артефактами в репозитории.

## Правила внедрения изменений

1. Любая задача начинается с краткой формулировки цели и критерия приемки.
2. Изменения выполняются минимальным диффом и обратимым способом.
3. Архитектурные решения принимаются только ролью planner/architect.
4. Реализация не меняет runtime-код вне согласованного scope.
5. Перед закрытием задачи обязателен блок валидации и перечень рисков.

## Критерии качества

- Изменение локализовано и не тянет лишних зависимостей.
- Нет смешивания UI-слоя с логикой трекинга.
- Нет смешивания бизнес-логики с hardware/IO адаптерами.
- Присутствуют проверяемые артефакты: diff, команды проверки, отчет.

## Операционный приоритет

1. Стабильность трекинга и предсказуемость lock-поведения.
2. Повторяемость training/evaluation цикла.
3. Упрощение операторского контура без потери контроля.
4. Подготовка к parity Mac ↔ RTX ↔ RPi/Hailo.
